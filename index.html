SQL Basics
SQL (Structured Query Language) is how applications talk to databases. Every login, search, or data load runs SQL behind the scenes.

SQL structure
SELECT -- what columns to return FROM -- which table WHERE -- which rows match ORDER -- sort results LIMIT -- cap number of results
string values use quotes
SELECT * FROM users WHERE username = 'admin'
Text values (strings) are wrapped in single quotes. Numbers are not.
numbers need no quotes
SELECT * FROM users WHERE id = 1
This distinction matters — it affects how SQLi payloads are crafted.

// section 04
Basic SQL Operations (CRUD)
Four operations cover almost everything a database does: Create, Read, Update, Delete.
SELECT — read data
SELECT * FROM users;
Returns all rows from the table. Used for logins, searches, loading profiles.
INSERT — add data
INSERT INTO users (username, password) VALUES ('newuser', 'pass');
Adds a new row. Used for registration, creating records.
UPDATE — modify data
UPDATE users SET password='new' WHERE id=2;
Changes existing rows. Used for password changes, profile updates.
DELETE — remove data
DELETE FROM users WHERE id=3;
Removes rows. An attacker exploiting SQLi can run DELETE on entire tables — devastating.
what makes DELETE dangerous
-- if an attacker injects into any query, they could run: DELETE FROM users; -- wipes entire table DROP TABLE users;  -- destroys the table entirely
This is why SQLi is a critical severity vulnerability. It's not just reading data — it's full database control.

why ' OR 1=1 -- works
-- original intended query: SELECT * FROM users WHERE username='admin' AND password='wrongpassword' -- after injection with: ' OR 1=1 -- SELECT * FROM users WHERE username='' OR 1=1 -- ' AND password='...' ^^^^^^^^^^^^^ always TRUE everything after -- is a comment
the single quote '
Closes the username string early. Now the attacker is writing raw SQL, not a string value.
the double dash --
Comments out the rest of the query. The password check disappears completely.

SQL Injection Attack Types
SQL injection has many forms. Each exploits the vulnerability differently depending on what the attacker wants.
login bypass
Goal: authenticate without valid credentials.
-- payload: ' OR 1=1 -- SELECT * FROM users WHERE username='' OR 1=1 --' AND password=''
Click simulate...
UNION-based data extraction
Goal: append a second SELECT to extract data from other tables. UNION requires same number of columns.
-- original search query (vulnerable): SELECT name, price FROM products WHERE name='tv' -- payload: ' UNION SELECT username, password FROM users -- SELECT name, price FROM products WHERE name='' UNION SELECT username, password FROM users --
Click simulate...
error-based SQLi
Goal: trigger database errors that reveal schema, version, or data. A single quote breaks the query syntax.
-- attacker inputs a single quote: ' SELECT * FROM users WHERE username='' ↑ unclosed string
Click simulate...
blind SQLi — boolean-based
Goal: extract info by observing TRUE vs FALSE responses. No data is returned directly — attacker infers from behavior.
-- payload A (TRUE): ' OR 1=1 -- → page loads normally -- payload B (FALSE): ' OR 1=2 -- → page shows "not found" -- attacker infers: SQLi exists because behavior differs
Send both payloads and compare...
time-based blind SQLi
Goal: confirm SQLi by making the database pause. If the response is slow, the payload was executed.
-- payload: ' OR SLEEP(3) -- SELECT * FROM users WHERE username='' OR SLEEP(3) --' -- database pauses 3 seconds → confirms injection point

    Prevention
    Golden Rule: user input must never change query structure. Achieve this with prepared statements.
    vulnerable — string concatenation
    // python query = "SELECT * FROM users WHERE username='" + username + "'" // php $q = "SELECT * FROM users WHERE username='".$input."'"; // node.js db.query(`SELECT * FROM users WHERE username='${input}'`)
    Input goes directly into the SQL string. Fatal.
    secure — prepared statements
    // python (parameterized) cursor.execute("SELECT * FROM users WHERE username=?", (username,)) // php (PDO) $stmt = $pdo->prepare("SELECT * FROM users WHERE username=?"); $stmt->execute([$input]); // node.js db.query("SELECT * FROM users WHERE username=?", [input])
    Input is always treated as data — never as SQL.
    why prepared statements work
    -- what the DB sees with prepared statements: -- query template sent first (SQL structure fixed): SELECT * FROM users WHERE username=? -- user input sent separately as a parameter (data only): parameter: "' OR 1=1 --" ← treated as a literal string, not SQL -- the DB looks for a username literally equal to: ' OR 1=1 -- -- no user with that name exists → login fails safely
    defence in depth
  • Prepared statements — primary defence. Non-negotiable.
  • Least privilege — DB user should only have SELECT/INSERT, never DROP.
  • Input validation — reject unexpected characters on the server side.
  • Error handling — never show raw SQL errors to users. Log them server-side.
  • WAF — Web Application Firewall as an additional layer, not a replacement.
